## 시작하기 전에

- 클라이언트와 서버는 요청, 응답 관계
- 앞서 나온 IP 주소 외에, 도메인 네임을 사용 가능
- 대표적인 식별자로 URL과 URN이 있음

## 도메인 네임과 네임 서버

IP 주소는 언제든지 바뀔 수 있고, 모든 호스트의 IP 주소를 기억하기 쉽지 않음

### 도메인 서버
도메인 서버는 도메인 네임에 대응하는 IP 주소를 관리하는 서버, DNS(Domain Name System) 서버

### 도메인 계층 구조
도메인 네임은 루트 도메인(.), 최상위 도메인(TLD com, kr, us 등), 2단계, 3단계, 4단계...등으로 이루어짐

도메인 네임을 전부 포함하는 도메인 네임을 전체 주소 도메인 네임(FQDN)이라고 부름

도메인 네임이 계층 구조를 이루는 것처럼, 도메인 서버도 계층 구조를 이루는데, 이렇게 계층적이고, 분산된 도메인 네임에 대한 관리 체계를 도메인 네임 시스템이라고 부름

+++ 도메인 네임 서버가 계층적인 이유는 인터넷의 수십억개의 도메인을 관심 분리, 책임 분리해서 관리하기 위함

## 계층적 네임 서버
네임 서버는
- 로컬 네임 서버
- 루트 네임 서버
- TLD 네임 서버
- 책임 네임 서버
의 계층 구조를 가짐

IP 주소를 알아내는 과정을 도메인 네임을 리졸빙하는 과정이라고 함

### 로컬 네임 서버
- 가장 먼저 찾는 네임 서버
- 일반적으로 ISP가 할당 or 공개 DNS(google 등)를 이용하기도 함

### 루트 네임 서버
- 루트 도메인을 관장하는 네임 서버
- 질의에 대해 대응하는 TLD 네임 서버의 IP 주소를 반환

### TLD 네임 서버
- TLD를 관리하는 네임 서버
- 질의에 대응하는 네임 서버 주소를 반환

### 책임 네임 서버
- 특정 도메인 영역을 관리(naver, google)
- 로컬 네임 서버가 마지막으로 질의하는 네임 서버, 최종 IP

### 재귀적 질의
- 클라이언트가 질의하면 재귀적으로 네임 서버를 타고 들어가서 질의

### 반복적 질의
- 로컬 네임 서버가 모든 네임 서버에 반복 질의

실제로는 네임 서버들이 응답 결과를 임시로 저장했다가 이를 활용하는데, 이를 DNS 캐시라고 부름(TTL동안 저장)

## 자원을 식별하는 URI

자원을 식별하는 정보를 URI라고 함.
자원을 이름으로 식별하는 것을 URN
위치를 통해 식별하는 것을 URL이라고 함
