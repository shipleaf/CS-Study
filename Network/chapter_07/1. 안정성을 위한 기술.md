## 시작하기 전에

서비스에서는 사용자 폭증을 대비한 안정성을 고려해야 한다.
서비스(서버)의 안정성을 확보하기 위해서는 물리적 장비나 프로그램 등을 여러 개 두는 **이중화**와 **다중화**,
트래픽을 고르게 분산하는 **로드 밸런서**를 알아야 한다.

## 가용성

**안정성**이란 특정 기능을 언제든 균일한 성능으로 수행할 수 있는 특성

안정성의 정도를 나타내는 용어로 가용성과 고가용성이 있다.

```
가용성 = 업타입 / 업타임 + 다운타임
```
으로 표현할 수 있고, 가용성의 수치가 높은 경우를 **고가용성(High Availability, HA)**이라고 한다.

일반적으로 안정적이라고 평가받는 시스템은 99.999% 이상을 목표로 한다. 이를 9가 '다섯 개'라는 의미에서 '파이브 나인스'라고 부른다.
이는 1년간 다운타임이 5.26분 정도에 그치는 수치이다.

가용성을 높이려면 다운타임을 최소화 해야 하는데, 서비스가 다운되는 원인은 매우 다양하다.
- 과도한 트래픽
- 예기치 못한 오류
- 하드웨어 장애
- 보안 공격
- 자연재해

이 모든 원인을 사전에 파악하고, 미리 방어하기란 쉽지 않다.<br>
따라서 문제가 발생하지 않도록 하는 것이 아니라, 문제가 발생하더라도 계속 기능할 수 있게 설계하는 것이 중요하다.<br>
이를 **결함 감내**라고 한다.

## 이중화

**이중화**란 '무언가'를 이중으로 두는 기술을 말한다.

여기서 '무언가'란 **결험 감내**를 위해 가용성을 높이기 위한 것으로, 
- 서버 컴퓨터
- 네트워크 인터페이스(NIC)
- 스위치
- 데이터베이스
- 웹 서버 프로그램

**단일 장애점**(SPoF; Single Point Of Failure)이란 문제가 발생할 경우 시스템 전체가 다운되는 지점을 말한다.
시스템의 고가용성을 위해 단일 장애점을 없애는 것이 바람직하다.

이중화 구성에는
- 엑티브/스탠바이
- 엑티브/엑티브

**다중화**는 말 그대로 두 개 이상으로 구성하는 것을 말한다.

**이중화/다중화**의 예시로 **티밍**과 **본딩**이 있는데, 전자는 윈도우, 후자는 리눅스에서 사용되는 용어
네트워크 인터페이스(NIC)를 이중화/다중화하여 더 뛰어나고 안정적인 성능을 보유한 하나의 인터페이스처럼 보이게 하는 기술이다.

## 로드 밸런싱

서버를 다중화 했더라도, **트래픽**의 분배가 없으면 말짱도루묵이다.

**트래픽**이란 '특정 시점에 노드를 지나는 네트워크 패킷의 양'으로, 트래픽이 과도하게 몰리면 서비스의 일관성을 유지하기 어렵다.

**트래픽**의 고른 분배를 위해 사용되는 기술이 **로드 밸런싱**('Load' + 'Balancing)이다.

**로드 밸런서**는 L7 스위치, L4 스위치 등 네트워크 장비로도 수행될 수 있지만, 로드 밸런싱 기능을 제공하는 소프트웨어를 설치하면 일반 호스트도 로드 밸런서로 사용할 수 있다.

대표적인 소프트웨어로 HAProxy, Envoy 등이 있고, 웹 서버 소프트웨어인 Nginx에도 로드 밸런싱 기능이 내장되어 있다.


대표적인 로드 밸런싱 알고리즘으로는,
- 라운드 로빈 알고리즘
- 최소 연결 알고리즘
- 가중치 라운드 로빈 알고리즘
- 가중치 최소 연결 알고리즘
등이 있다.

++ 로드밸런서로써의 Nginx
```
upstream backend_servers {
    least_conn;
    server backend1:8080 max_fails=3 fail_timeout=10s;
    server backend2:8080 max_fails=3 fail_timeout=10s;
}
```
Nginx는 다른 알고리즘을 지원하지만, 기본적으로 **라운드 로빈** 방식으로 로드 밸런싱을 수행한다.

위 설정은
```
10초 동안 3번 실패하면
그 서버를 잠시 장애 서버로 보고 제외
```
를 의미한다. 단, 기본적으로 Nginx의 health check는 passive에 가깝기 때문에, 실제 요청을 health check 용으로 쓴다.
해서, 더 안정적인 health check를 수행하려면 별도의 외부 도구를 사용해야 한다.
